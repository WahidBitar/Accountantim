# Story 4.13: k6 public-statement throughput harness

Status: ready-for-dev

## Story

As **Wahid**,
I want **a k6 scenario sustaining 166 rps against the public-statement route for 10 minutes**,
so that **NFR-SC4 (83 rps PRD commitment) has 2× headroom proven and AR-044 rehearsal #5 is executable** (Murat's budget).

**Covers requirements:** Epic 4 — NFR-SC4 (83 rps sustained), §2.10 pre-launch rehearsal #5, AR-044 [Source: epics.md Epic 4 Story 4.13; architecture.md §2.10].

**Why this story now:** Story 4.5 ships the view endpoint. Story 4.9 rate-limits it. Story 4.11 caps costs. This story proves the topology survives 2× PRD-committed load without melting. Runs as part of release-candidate acceptance.

## Acceptance Criteria

### AC1 — `tests/Load/public-statement-throughput.js` k6 script

**Given** k6 is the designated load-test tool
**When** the script is authored
**Then** `tests/Load/public-statement-throughput.js` exists with the following skeleton:

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Counter, Rate, Trend } from 'k6/metrics';

export const options = {
  scenarios: {
    sustained: {
      executor: 'constant-arrival-rate',
      rate: 166,
      timeUnit: '1s',
      duration: '10m',
      preAllocatedVUs: 50,
      maxVUs: 100,
    },
  },
  thresholds: {
    'http_req_failed{expected_response:true}': ['rate<0.001'],  // error rate < 0.1%
    'http_req_duration{expected_response:true}': ['p(95)<300', 'p(99)<1500'],
  },
  summaryTrendStats: ['p(95)', 'p(99)', 'avg', 'max'],
};

const tokens = JSON.parse(open('./tokens.json'));  // pre-seeded
const bypassToken = __ENV.FAKTUBOH_RATE_LIMIT_BYPASS_TOKEN;

export default function () {
  const token = tokens[Math.floor(Math.random() * tokens.length)];
  const res = http.get(`${__ENV.TARGET}/v1/statements/t/${token}`, {
    headers: {
      'X-Faktuboh-Bypass': bypassToken,
      'Accept-Language': Math.random() > 0.5 ? 'ar' : 'en',
    },
  });
  check(res, {
    'status is 200': (r) => r.status === 200,
    'body has balances': (r) => r.json('balances') !== undefined,
  });
}
```

**And** `tokens.json` is generated by a seed script that creates N test owners + contacts + statements + tokens [Source: epics.md Story 4.13 AC].

### AC2 — Pass thresholds: p95 < 300ms, p99 < 1.5s, error rate < 0.1%

**Given** NFR-SC4 operational targets
**When** the script runs against staging (or a dedicated load-test environment)
**Then** k6 summary reports all thresholds PASSED:
- `p(95)` of `http_req_duration` < 300 ms
- `p(99)` of `http_req_duration` < 1500 ms
- Error rate < 0.1% (1 error per 1000 requests)
- Zero 5xx responses (separate check: `status_is_2xx` rate == 1.0 excluding rate-limit)
**And** any threshold breach exits with non-zero status, failing the CI workflow
**And** k6's summary output is captured to `summary.json` + posted to the dashboard [Source: epics.md Story 4.13; architecture.md §4.6 D4.19].

### AC3 — Sustains 166 rps for full 10 minutes

**Given** ramp-up matters for cache warm-up
**When** the test runs
**Then** the `constant-arrival-rate` executor guarantees exactly 166 rps (not "as fast as possible")
**And** the 10-minute duration produces 99,600 total requests
**And** the script includes a 30-second warm-up phase BEFORE the measured window (pre-allocates VUs, triggers cache warm-up with unmeasured requests)
**And** if the target cannot sustain 166 rps (k6 metric `dropped_iterations` > 0), the test FAILS [Source: epics.md Story 4.13].

### AC4 — AFD/SWA cache-hit ratio > 95%

**Given** the public-statement route is cache-heavy per §4.6 D4.19
**When** the test runs
**Then** the cache-hit ratio is captured from the CDN's telemetry (SWA metrics or AFD access log)
**And** ratio must be > 95% across the 10-minute window
**And** if < 95%, the test fails with diagnostic output indicating which cache-miss tokens were hit most often
**And** the script includes a finite token pool (e.g., 20 tokens × 5k requests each) to enable cache-hit dominance [Source: architecture.md §4.6 D4.19; §4.9 silent-decay SLOs].

### AC5 — Results post to Golden-signals dashboard

**Given** load-test results should be trended
**When** the k6 run completes
**Then** the CI workflow posts results to App Insights custom events:
- `load_test_run` with properties `{ scenario: 'sustained', rps: 166, p95_ms, p99_ms, error_rate, cache_hit_ratio, pass: true/false }`
**And** the Golden-signals dashboard trends `load_test_run` across weeks
**And** a panel shows historical pass/fail for each release [Source: architecture.md §4.6 D4.20].

### AC6 — Release-candidate acceptance suite

**Given** this is a gate for production deploys
**When** `deploy-prod.yml` is updated
**Then** the production deploy workflow:
1. Deploys to staging first
2. Runs `tests/Load/public-statement-throughput.js` against staging
3. Waits for pass before allowing the prod deploy step
**And** the gate is manually bypassable in emergencies (recovery deploys) with an approval + audit log entry [Source: epics.md Story 4.13; architecture.md §4.6 D4.17].

### AC7 — Seed script produces `tokens.json`

**Given** load tests need pre-seeded valid tokens
**When** `scripts/load-test/seed-tokens.mjs` runs
**Then** it:
1. Authenticates as a test user (OIDC service account or Auth0 M2M client in staging)
2. Creates N test contacts + debts + statements
3. Collects the issued tokens
4. Writes to `tests/Load/tokens.json`
**And** the seed script is idempotent (re-runs clean up previous seed data first)
**And** the seed token count is 20 by default (configurable via `--count <N>`) — 20 gives enough variety for cache-key analysis while keeping setup fast [Source: epics.md Story 4.13 AC; architecture.md §7.5.8 seed-data].

### AC8 — Bypass token is required; test refuses without it

**Given** rate limits from Story 4.9 would catastrophically skew results
**When** the harness starts
**Then** the script checks `__ENV.FAKTUBOH_RATE_LIMIT_BYPASS_TOKEN`
**And** if missing, exits with clear error: `"FAKTUBOH_RATE_LIMIT_BYPASS_TOKEN is required. See docs/ops/load-test-bypass-token.md"`
**And** the bypass token is a GitHub Actions secret; rotated monthly [Source: Story 4.9 AC8].

### AC9 — Cost cap + emergency shutdown not triggered

**Given** load tests should not trigger AR-050 cost cap (Story 4.11)
**When** 99,600 requests hit
**Then** no token exceeds 1000 views/24h cap (the test pool of 20 tokens is sized to spread traffic: 99,600/20 = 4,980 views per token — EXCEEDS 1000, so test must use ≥100 tokens OR the test must temporarily elevate the cap via `FAKTUBOH_DAILY_VIEW_CAP=10000` env override in staging)
**And** the staging env applies the elevated cap via `appsettings.Staging.json` override
**And** `FAKTUBOH_EMERGENCY_SHUTDOWN` is false during test runs (verified as a precondition check) [Source: Story 4.11 AC].

### AC10 — Baseline variant: 83 rps acceptance test

**Given** NFR-SC4 is the PRD commitment (not the 166 rps headroom target)
**When** a second k6 scenario runs
**Then** `tests/Load/public-statement-nfr-sc4-baseline.js` sustains 83 rps for 10 minutes with same thresholds
**And** both scenarios (baseline + headroom) run in the release-candidate suite
**And** if the headroom run fails but the baseline passes, the release is marked "acceptable but at risk — file revisit" [Source: epics.md Story 4.13; architecture.md §2.10 — PRD commitment is 83 rps; load test floor is 166 rps].

## Dev Notes

### Critical guardrails

1. **`constant-arrival-rate` executor.** The `ramping-arrival-rate` executor ramps up and finishes when requests complete — the load is NOT guaranteed to reach 166 rps. Pin to `constant-arrival-rate`.
2. **Staging, not prod.** Load tests against prod would trigger cost cap + alert storms. Only staging.
3. **Bypass token required.** Without it, 429s from rate limits dominate the results. The script must refuse to start.
4. **Cap override in staging.** Test runs will exceed 1000 views/token in 10 minutes; elevate the cap to 10k in staging only.
5. **Warmup phase before measurement.** Cold-start cache = poor p99. Pre-warm with 30s of unmeasured traffic.
6. **Token pool size matters for cache-hit ratio.** Too few tokens = 100% cache hit (unrealistic); too many = 80% cache hit (degrades throughput). 100 tokens is the calibrated middle.
7. **Don't run in prod.** Paper the runbook with "NEVER run against prod" warnings.
